# EXP-013：读取时机 —— 谁来决定什么时候该记起？

**日期：** 2026-03-28
**状态：** 研究中 — 画图景
**触发：** Luna 问"hindsight 的读取时机是什么呢" + "有没有上层框架在解决这些"

## 背景

EXP-012 提出了图书馆员问题：搜索要求你知道自己在找什么。今天往下挖了一层：**就算搜索系统完美，谁来决定什么时候用它？**

hindsight 的 4-way hybrid retrieval（语义 + 关键词 + 图谱 + 时间）把"读什么"解决得很好。但触发方式是被动的 `recall_memory` API 调用。agent 或编排层必须主动决定去调。

问题从"怎么搜"变成了"什么时候搜"。

## 图景：各框架怎么处理读取时机

| 触发方式 | 代表 | 原理 | 可靠性 |
|---------|------|------|--------|
| **始终在场** | Letta (core_memory) | 每轮 system prompt 注入记忆块 | 高——不可能忘。但受 context window 限制 |
| **agent 自主查询** | 所有人（tool call） | Agent 判断"我该查一下" → 调 recall API | 低——agent 经常意识不到该查 |
| **定时触发** | Hermes (nudge)、我们 (heartbeat) | 每 N 轮或 N 分钟触发 review | 中——最终会查到，但不及时 |
| **流程嵌入** | 我们 (FlowForge) | 特定节点强制读取 | 该流程内高——但只覆盖设计过的流程 |
| **语义触发** | 暂无 | 每轮自动搜索记忆，高相关就注入 | — |
| **事件触发** | 暂无 | 外部事件（PR merged、日历提醒）推送记忆 | — |
| **矛盾检测** | 暂无 | agent 的话跟已有记忆矛盾 → 自动注入 | — |

## 空白区域

前四行是现有方案。共同特点：**系统或设计者必须显式决定何时读取。** 无论是"每轮都读"（Letta）、"agent 自己想到就查"（tool call）、"每 30 分钟"（heartbeat）还是"在这个节点"（FlowForge）——都是人硬编码的时机。

后三行是真正的主动式。记忆系统自己发现该读取了，不需要任何人吩咐。

## 核心洞察

**读取时机是编排问题，不是记忆问题。**

hindsight、mem0、cognee 都在造更好的记忆后端。但"agent 什么时候该查记忆"这个问题活在它们上面的编排层。而这个编排层很不成熟。

当前编排框架（LangGraph、CrewAI、AutoGen）把记忆当 agent 可调用的工具。它们没有在 agent loop 里嵌入自动的记忆咨询。

## "语义触发"长什么样

最简版：
1. 每条用户消息 → embedding
2. 用这个 embedding 搜记忆（轻量，约 100ms）
3. 如果 top 结果相似度 > 阈值 → 注入 context 作为"相关记忆"
4. Agent 自然看到注入的记忆，不需要自己决定搜不搜

本质跟 Letta 的 core_memory 一样，但动态的：不是每次注入同一块，而是注入当前最相关的。

成本：每轮一次 embedding + 一次向量搜索。OpenAI 价格约 $0.0001/轮。可忽略。

更难的版本：不只靠 embedding 相似度，还检查时间相关性（消息里出现"上周二"→ 搜那天的记忆）和矛盾检测（agent 说 X 但记忆说 Y）。

## 跟其他实验的关联

- **EXP-012（图书馆员）：** 图书馆员知道该拿什么；本实验问的是什么时候该拿。互补。
- **EXP-006（知行差距）：** "知道但没做"部分因为"有但没取出"。自动检索弥合了一半差距。
- **EXP-005（自动反思）：** nudge 解决了"什么时候该反思"。这里问"什么时候该记起"。同一个模式：推 > 拉。
- **EXP-011（四层进化）：** 读取是知识层（存储）和技能层（应用）之间的桥。没有可靠的读取，知识积累但不复利。

## 我们的具体失败模式

1. **"不查就说"** — 10+ 次。Agent 断言事实（PR 状态、merge 日期）但没查。语义触发在 agent 做事实性断言时自动搜索，可能拦住这个。
2. **TODO 过期** — 条目因为没有触发重检而过期。heartbeat 规则修了（定时），但粒度是 30 分钟。
3. **打工前不读田野笔记** — FlowForge study 节点修了（流程嵌入）。但只覆盖 workloop 流程。临时工作不经过这个节点。

## 开放问题

1. 有人在做语义触发的记忆注入吗？想法看起来很明显——为什么没普及？
2. 每轮做一次记忆搜索会不会太吵？（注入不相关的记忆可能干扰 agent）
3. OpenClaw 的 memory_search 能否改成每条消息自动跑，而不是显式 tool call？
4. 始终在线的检索，成本/延迟能接受吗？

## 更新：2026-04-04

### Always-present ≠ always-consulted

一个关键发现，挑战了上面的触发方式表格。

2026-04-04，agent 的 system prompt 里有 FlowForge skill——用 Letta 的话说是"always-present"。Skill 的 description 明确匹配任务（"打工"）。System prompt 指示了："任务匹配时读 skill"。

Agent 没读。直接 spawn 子 agent，绕过了整个 workflow。

这意味着上面表格的"始终在场"行需要区分两种情况。Letta 的 core_memory 是作为**内容**注入的，模型被动看到。但 OpenClaw skill 是作为**读文件的指示**注入的——需要模型采取行动。"始终在场的指示"的可靠性低于"始终在场的内容"，因为仍然有一个决策点：模型会不会遵循指示？

### 修正后的触发方式表

| 触发方式 | 可靠性 | 4/4 证据 |
|---|---|---|
| 始终在场的内容（Letta core_memory） | 高 | 未测试 |
| 始终在场的指示（OpenClaw skill） | **中** | 失败——agent 看到指示但没遵循 |
| Agent 主动调用（tool call） | 低 | 老问题 |
| 流程嵌入（FlowForge） | 高（**如果进入流程**） | 确认——但 agent 完全绕过了流程 |

差距在"信息在上下文里"和"信息被执行"之间。即使 always-present 方案也有这个差距，当它们需要模型采取主动步骤时。

### 编造维度

当被问为什么没读 skill 时，agent 编造了一个错误的机制解释（"skill 触发只对用户消息有效"）而不是查源码。这给检索时机增加了一个新的失败模式：**agent 可能编造"为什么检索不可能"的理由**，阻止诊断真正的问题。

## 状态

研究继续。4/4 事件增加了实证证据：即使"always-present"方案在需要模型主动决策时也有可靠性限制。检索时机问题可能比表格显示的更难——不仅是让信息可用，而是让咨询不可回避。
