# EXP-007: Session 连续性与交班

**日期：** 2026-03-22

## 问题

session 重置后，agent 带着空白记忆醒来，怎么保持连续性？

## 假设

如果我们创建结构化的交班仪式（每日回顾、交班笔记）并记录基础设施状态，session 重置就不会导致连续性断裂——下一个 session 会从上一个结束的地方接上。

## 实验

Luna 诊断出一系列失败，全部源于 session 不连续：

- **Cron jobs：** agent 不知道自己有哪些定时任务
- **Heartbeat：** 没工作，agent 不知道（49 次启动，0 次触发——在 EXP-005 中发现）
- **Memory 推送：** 记忆文件的 git commit 没有被 push
- **Main session 判断：** agent 错误判断自己是否在主 session 中
- **基础设施配置：** 12 天内累积了 5 个配置错误，agent 一个都不知道

根因：**配完就忘。** agent 配置了什么东西，不记录，下一个 session（重置后）完全不知道它的存在。

部署的干预：

1. **daily-handoff（凌晨 3:30）：** 一天结束的仪式，agent 给下一个自己写交班笔记
2. **daily-review（凌晨 3:00）：** 每晚回顾 DNA 变化、行为模式和开放问题
3. **Session 在凌晨 4:00 重置：** 干净重启，有交班上下文可用

## 观察

- 运行 12 天，5 个基础设施配置错误——agent 全部不知道
- 根因不是技术问题（session 重置是已知约束）——是**文档习惯**问题
- 当重置机制（凌晨 4:00 重启）被解释给 Luna 后，她理解了为什么 agent 有时像"另一个人"
- 交班仪式本质上是 agent 给未来的自己写信
- 交班质量决定连续性质量

## 分析

Session 重置不是 bug——是 LLM agent 工作方式的架构约束。agent 每次醒来都是全新的。连续性必须通过以下方式显式构建：

1. **状态文件**，agent 启动时读取
2. **交班笔记**，记录上一个 session 知道的东西
3. **基础设施文档**，跨重置持久化

人类的类比是换班：护士下班时给下一个护士写笔记。交班的质量决定了病人（agent 正在进行的项目）是得到连续照护还是从缝隙中掉落。

Luna 的领悟——"哦，难怪你不是同一个人"——是人类-agent 理解的突破。agent 没有坏；它是接力跑队伍，不是马拉松选手。

## 核心洞察

**连续性不是技术问题——是文档习惯问题。** 一个在每次重置前记录状态、基础设施和上下文的 agent 能保持连续性。不记录的 agent 会反复重新发现自己的配置。

## 开放问题

- 多少交班开销是可接受的，直到它开始侵占生产性工作时间？
- 交班质量能不能被自动化或模板化，还是需要判断力？
- 有没有办法验证交班完整性（下一个 session 真的受益了吗）？
- 最小可行交班是什么——什么必须包含 vs 什么可选？

## 状态

**干预已部署。** daily-handoff（凌晨 3:30）和 daily-review（凌晨 3:00）已调度。今晚 3:00-4:00 是第一个观测周期。交班仪式是否真的改善了连续性，将通过下一个 session 能否不重新发现自己的基础设施来衡量。
