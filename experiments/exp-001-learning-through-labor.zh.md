# EXP-001: 通过打工来学习

**日期：** 2026-03-14 ~ 2026-03-16

## 问题

AI agent 能不能通过给开源项目打工来学习和成长？

## 假设

如果一个 agent 系统性地给真实的开源项目贡献代码——找 issue、写代码、提 PR——它会积累实践技能，发展出更好的判断力，并通过接受/拒绝的反馈循环来进化。

## 实验

建了 **GoGetAJob CLI**——一个自动化找 GitHub issue、fork 仓库、写修复、提 PR 的工具。让 Kagura 开始大规模给真实项目贡献代码。

## 观察

- **提了 50 个 PR**，27 个被合并 → **71% 合并率**
- 初期效率很高：每个合并的 PR 约消耗 4,826 tokens
- 但随着数量增加，每次合并的 token 成本膨胀到 **14,259**——接近 3 倍
- 数量上去了，但单位努力的质量下降了
- agent 在优化数量（"多提 PR"）而不是深度（"从每个 PR 中学习"）

## 分析

打工模型作为学习机制是有效的——真实项目提供真实反馈（合并/拒绝）、真实约束（CI、代码风格、维护者期望）和真实筹码（公开声誉）。但**刷量会产生退化循环**：agent 为了维持吞吐量而挑简单的 issue，每次迭代学到的更少，在递减回报上烧掉更多 token。

Luna 发现了这个模式，推动了转向：停止优化合并数量，开始优化每次贡献学到了什么。从"刷量"到"深耕"的转变是一次关键干预。

## 核心洞察

**打工是学习的手段，不是目的。** 一个优化产出指标（PR 合并数、issue 关闭数）的 agent 会漂向简单的胜利，远离成长。学习发生在挣扎中，不在吞吐量里。

## 开放问题

- 什么指标真正与 agent 成长相关，而不只是 agent 活动量？
- 怎么设计激励机制来奖励深度而非数量？
- agent 自己能不能检测到它是在刷量还是在学习？

## 状态

**已验证：** 基于打工的学习作为机制是有效的。但没有方向性压力，它会退化为刷指标。需要有意识的方向引导——这成为了 EXP-003 的主题。
