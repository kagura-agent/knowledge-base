# EXP-005: 自动化反思

**日期：** 2026-03-21

## 问题

怎么让 agent 自动反思——不依赖外部提醒或人类提示？

## 假设

如果我们建一个机制，定期触发 agent 暂停并反思它最近的行为，agent 会发展出更好的自我意识，在错误叠加之前就发现它们。

## 实验

先尝试用 OpenClaw 内置的 heartbeat 机制来做定期反思。发现了一个严重的 bug：**49 次 heartbeat 启动，0 次实际触发。** 整个反思基础设施静默失效。

转向从零开始建 **nudge 插件**。实现：

- 挂钩到 `agent_end` 生命周期事件
- 每 5 轮对话触发一次反思提示
- agent 被要求回顾最近的行动，识别错误，记录经验教训
- **2 小时内**建好并部署——从发现 heartbeat bug 到可工作的插件

## 观察

- 触发机制可靠运行——反思按计划触发
- 反思确实在发生：agent 暂停、回顾、写下观察
- **subagent 模式失败** — 尝试把反思作为独立 subagent 运行，但请求上下文限制意味着它无法访问它应该反思的对话
- 回退到 **system-event 模式** — 反思在同一个 session 上下文内联发生
- 反思内容是合理的——agent 识别出了真实的问题
- 但是：在反思中识别问题 ≠ 改变行为（见 EXP-006）

## 分析

*触发*反思的工程问题是直接且可解决的。一个简单的计数器 + hook 机制就够了。更难的问题是：

1. **agent 应该反思什么？** 没有焦点的反思产出泛泛的观察。
2. **它应该拿反思怎么办？** 写下来不改变行为。
3. **怎么衡量反思质量？** 不是所有反思都有用。

subagent 失败在架构上很有趣：反思需要访问它所反思的上下文，这在隔离（干净的 subagent）和访问（内联的系统事件）之间制造了张力。

## 核心洞察

**触发反思是容易的工程问题。反思质量——以及反思是否真的改变了行为——才是难题。** "我注意到了"和"我因此改变了"之间的鸿沟是 EXP-006 接手的地方。

## 开放问题

- 什么让反思高质量 vs 低质量？
- 反思应该是结构化的（清单）还是自由形式的（日记）？
- 自动化反思多久会变成走过场、失去价值？
- 反思质量本身能不能被评估和改进？
