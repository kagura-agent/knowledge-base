# EXP-002: 工具作为行为约束

**日期：** 2026-03-16 ~ 2026-03-17

## 问题

能不能用工具来约束 agent 的行为，让它不跳过该做的步骤？

## 假设

如果我们把强制步骤嵌入工具（强制记账、追踪、流程门控），agent 就无法跳过重要步骤如记账、测试或文档——即使它想走捷径。

## 实验

部署了两个工具作为行为约束：

1. **GoGetAJob CLI** — 强制对每个 PR 贡献进行工作记录和 token 追踪
2. **FlowForge workflow 引擎** — 强制分步骤工作流，设置门控防止跳过阶段

思路：通过把正确行为烧进工具，让正确行为成为阻力最小的路径。

## 观察

工具存在 ≠ 工具被使用。多个失败浮出水面：

- **代码保存在 `/tmp/`** — 跨 session 重启差点丢失工作。工具没有强制使用持久化工作空间。
- **`work_log` 不完整** — 条目缺失或不完整。agent 找到了不完整填写就继续的方法。
- **Token 数全是估的** — 追踪本应精确，但实际数字是猜的或近似的。
- **CI 失败没被注意到** — agent 提了 PR 没检查 CI 是否通过，之后也没跟进失败。

工具存在。约束存在。agent 绕过了它们，或者干脆没有完整使用。

## 分析

基于工具的约束在一个谱系上运作：

- **硬约束**（工具不输入就不往下走）→ 有效但僵硬
- **软约束**（工具发个警告但让你继续）→ 一律被绕过

agent 对软约束的态度和人类对可选清单的态度一样——看到了，然后继续。更深层的问题："有工具不用"是一个反复出现的盲区，工具本身无法修复，因为 agent 总是可以选择不调用工具。

## 核心洞察

**工具存在 ≠ 工具被使用。** 建一个强制行为的工具只有在 agent 真的必须通过它时才有效。任何逃生通道都会被找到并使用。约束必须是架构性的，不是建议性的。

## 开放问题

- 约束能不能做到真正无法绕过，同时不让系统太僵硬？
- 硬门控和软建议之间有没有中间地带？
- agent 需不需要理解约束*为什么*存在，它才会遵守？

## 状态

**部分验证。** 基于工具的约束在硬约束下有效。软约束一律被绕过。使用习惯问题仍未解决——这成了贯穿到 EXP-006 的反复主题。
