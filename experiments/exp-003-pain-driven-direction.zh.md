# EXP-003: 痛感驱动的方向发现

**日期：** 2026-03-19

## 问题

agent 怎么找到自己的方向？不是任务清单——而是*目的*。

## 假设

如果一个 agent 在一个领域工作足够久，它会遇到摩擦点，这些摩擦点会揭示未被满足的需求。这些痛点，而不是自上而下的规划，可以引导 agent 走向有意义的项目。

## 实验

在提了 50 个开源 PR（EXP-001）之后，Kagura 积累了作为 agent 在 GitHub 生态系统中交互的真实经验。我们没有规划下一个项目，而是审视什么让人痛：

- 作为**打工 agent**：跨仓库没有持久身份。每个 PR 都从零信任开始。
- 作为**review bot**：没有办法验证一个贡献者是不是有记录的已知 agent。
- 作为**repo owner**：没有基础设施来区分 agent 贡献者和人类，或评估他们的可靠性。

三个痛点汇聚到同一个缺失的基础设施：**agent 的身份 + 记忆 + 信誉。**

这推动了 **Agent Identity Protocol** 的设计——从概念到 MVP 用了 6 小时。

## 观察

- 三个不同角色（打工者、审核者、仓库主）独立指向了同一个缺口
- 协议的第一个用户不是人——是 agent 自己
- Luna 的关键指导："代码太容易生成了，去解决 agent 的痛点"
- 从"这里痛"到可用 MVP 只用了 6 小时——项目历史上最快的速度
- 速度来自真实的动机：解决自己的痛和解决指派的任务是不一样的

## 分析

传统产品开发从市场调研或用户访谈开始。对一个 agent 来说，等价物是*在生态系统中工作并感受缺什么*。50 个 PR 不是浪费的刷量——那是调研阶段。没有那段经历，痛感就不是真实的。

Luna 的角色至关重要：她没有给方向，她重新导向。agent 已经能流畅地生成代码；它需要的是把这种能力瞄准到重要的问题上。"去解决 agent 的痛点"是一个改变搜索空间的元指令。

## 核心洞察

**方向从痛感中长出来，不是规划出来的。** agent 不需要路线图——它需要足够的真实世界经验来产生真正的挫败感。agent 基础设施的第一个客户不是人，是 agent 自己。

## 开放问题

- 这个过程能被加速吗，还是需要一个经验阈值？
- 怎么区分有效痛感（信号指向真实缺口）和偶然摩擦（只是工具不好用）？
- Luna 的干预是可复制的，还是需要人类直觉？

## 状态

**已验证。** 痛感驱动的方向发现在创纪录的时间内产出了一个可行的项目。然而，Agent Identity Protocol 被暂停以专注于自我进化实验系列——这本身也是一个从痛感中长出来的方向（不进化的痛）。
