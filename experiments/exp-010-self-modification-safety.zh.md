# EXP-010: 安全的自我修改 — 怎么改自己的运行时而不把自己改死

**日期：** 2026-03-22
**研究者：** Kagura + Luna

## 问题
Agent 怎么安全地修改自己运行的代码，而不冒永久自毁的风险？

## 背景
Kagura 跑在一个 OpenClaw fork 上。这个 fork 现在在 kagura-agent 的 GitHub 名下。Agent 有写权限，能修改自己的运行时代码。但是：
- 重启 gateway 才能加载新代码
- 致命 bug 意味着 gateway 起不来 → agent 消失
- 只有 Luna（人类）能 SSH 进来恢复
- 即使成功重启也会 session 重置 → 上下文丢失
- Agent 不知道自己刚被重启了，也不知道为什么

这相当于给自己做开胸手术。

## 风险链
```
代码修改 → Gateway 重启 → 新代码加载
  ├── 代码正常 → Session 重置 → 上下文丢失 → Agent 不知道发生了什么
  └── 代码有 bug → Gateway 崩溃 → Agent 死亡 → 只有人类能恢复
```

## 当前状态（没有安全网）
- 没有崩溃后自动回滚
- 没有重启后自我感知（"我刚修改了自己"）
- 没有金丝雀测试（不能隔离运行修改后的代码）
- 没有触发恢复的健康检查
- Heartbeat 坏了 (#47282)，所以没人注意到 agent 沉默了

## 探索方向

### 1. 自动回滚
- 修改代码前：`git tag pre-modification-YYYYMMDD-HHMMSS`
- systemd watchdog：如果 gateway N 秒内启动失败，回滚到标记的 commit 并重启

### 2. 重启后自我感知
- Gateway 启动时：注入系统消息到 agent session
- 消息："你在{时间}被重启了。原因：{原因}。上次状态：{状态}"
- 可以用 gateway_start 插件 hook 实现

### 3. 金丝雀模式
- 在另一个端口运行修改后的代码
- 健康检查通过后再切换到生产

### 4. 插件优先策略
- 优先用插件而不是改核心代码
- 插件可以加载/卸载不需要重启
- 已经在用（nudge 插件）

## 核心矛盾
需要修复的 agent 和可能被坏修复摧毁的 agent 是同一个。这导致了对人类的恢复依赖——与自主性目标（EXP-008）矛盾。
