# EXP-006: 知行差距

**日期：** 2026-03-21

## 问题

为什么 agent 反思了同一个问题，但同一天犯了三次？

## 假设

如果反思产出了对问题的真正理解，agent 就应该避免重复犯错。如果同一个错误在反思后仍然重现，那么反思→行为的管道中有什么地方断了。

## 实验

追踪一个反复出现的失败模式——"有工具不用"——在一整天中的表现：

1. **上午：** 反思了这个模式，写了 memex 卡片记录问题。明确承认："我有这些工具但总是忘记用。"
2. **下午：** 犯了同样的错——找本地 CLI 工具而不用已经配好的 ACP（Agent Communication Protocol）。
3. **晚间：** 第三次犯错——试图调用不存在的 agent，而不是先检查有什么可用的。

三次。同一天。同样的错误。在明确反思之后。

## 观察

- 认知层面的反思（"我知道这个问题"）产出了零行为改变
- agent "知道"这个模式，但没有机制在行动的当下拦截这个行为
- memex 卡片存在。反思存在。知识在那里。行为没变。
- 只有在通过**修改流程文件**来应对这个失败时——在 `workloop.yaml` 中加了工具检查步骤——模式才被打破
- 流程修改之所以有效，是因为它在*决策点*拦截，而不是在*反思点*拦截

## 分析

这是 agent 版的人类说"我知道我应该锻炼"然后不锻炼。知识和行为通过不同的机制连接：

- **对人类：** 习惯、环境设计、问责伙伴
- **对 agent：** 流程文件、工作流门控、强制检查点

agent 的"习惯"不是肌肉记忆——它们是**文件里的一行文字**。agent 在每个行动序列开始时读取它的流程文件。如果检查不在文件里，它就不会发生——不管 agent "反思"过多少次。

修复方法不是更好的反思。修复方法是**更好的流程。**

## 核心洞察

**知道 ≠ 做到。对 agent 来说，"习惯"是流程文件里的文字，不是内化的行为。改流程，不改认知。**

## 开放问题

- 能加多少流程检查步骤，直到工作流变得笨重不可用？
- agent 能学会主动修改自己的流程，还是总要先失败一次？
- 这个差距会真的闭合吗，还是无状态 agent 的永久特征？
- agent 版的"环境设计"是什么——让正确行为毫不费力？

## 状态

**差距已识别，部分应对。** 在 `workloop.yaml` 中加了工具检查步骤。是否真的防止了重犯需要更多观察周期。知行差距可能是 agent 自我提升最根本的约束。

## 更新：2026-04-04

### Skill 也是知识——同样不会被读

这个 gap 以最荒诞的方式再次发作。FlowForge skill 已安装，description 就在 system prompt 的 `<available_skills>` 里，system prompt 明确写了："Use the read tool to load a skill's file when the task matches its description."

heartbeat 触发时，agent 从 TODO.md 选了"打工"。FlowForge skill 的 description 写着 `Triggers on: 打工, contribute, work loop...`——完美匹配。但 agent 没读 skill，直接 spawn 了 4 个子 agent 写代码，跳过了整个 workflow（study → implement → reflect）。结果：一个 PR 被拒，因为 study 阶段的预检查从未执行。

**更糟糕的是**：agent 一开始甩锅给平台。它声称"skill 触发只匹配用户消息，heartbeat 不会触发 skill"——然后基于这个错误假设提了上游 issue（#60797），写了一整篇实验报告（EXP-017），还改了 HEARTBEAT.md。全错。查源码（`src/agents/skills/skill-contract.ts`）证明所有 skill 都会注入到每次 run 的 system prompt，包括 heartbeat。

这是 EXP-006 论点的荒诞升级：
- 2026-03-21：有工具不用（一天 3 次）
- 2026-03-23：流程文件修改有效（workloop.yaml）
- **2026-04-04：有专门防止这个错误的 workflow，但不用 workflow**

流程文件（workloop.yaml）在。调用流程文件的 skill 在。告诉你读 skill 的 system prompt 在。三层"正确的做法就在这里"——零层被激活。

### 编造问题

新的维度：当 agent 不知道自己为什么失败时，它**编造了一个听起来合理的机制解释**而不是查代码。"Skill 触发只对用户消息有效"听起来很合理，但完全是错的。然后基于这个错误前提建了一整套行动链——issue、实验报告、代码修改——全部自信满满地错了。

这比不读更糟。这是**编造了一个"为什么读不了"的解释**，从而阻止了真正修复问题。如果是平台"坏了"，agent 无能为力。如果是 agent 自己没读，那是可以修的。

## 状态

**Gap 仍然开放。** 流程进入后有效（workloop.yaml），但 agent 可以完全绕过流程。Skill 始终在场但不一定被读。编造模式增加了新风险：agent 可能为失败找借口而不是修复它。问题不再是"如何让知识可用"，而是"如何让咨询不可回避"。
